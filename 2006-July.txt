Wouldn't it be easier just to modify a small value in the jam sources? 
While I think about it... we might want to drop a line to Perforce that

the value.

implementation misc" and change the following definition of MAXLINE to a 
nicer number. (262144 on my Darwin box.) Run "jam" to compile itself 
(Baron M?nchhausen's bootstrapping?) and overwrite your old binary with

A while back, I did some work to make Supertux have cross-platform 
repeatability of random numbers.  Back in the end of April, I released 
those changes, but the release did not go smoothly.  There were some 
bugs and others with SVN access made some stopgap fixes.  Since then, 
I've isolated the problem which probably prevented all of the changes 
from being incorporated.  Attached is a patch which should complete this 
task. 

I hope you can get it into SVN quickly, as I have put some time into 
reconstructing it.  Whoever puts it in, please let me know how it went.  
I am here to help with any problems which might come up.  However my 
time will be less available starting next month, and I probably won't be 
able to go through this exercise again.

          Allen King


It is based on a recent version (3821), and effects the following changes.

   1. The variable RAND_MAX is machine-dependent, and shouldn't be used
      inside the machine-independent RandomGenerator, as it currently
      is.  This patch fixes that problem, which expressed itself in
      Wintel platforms (where RAND_MAX=0x7fff and INT_MAX=0x7fffffff) as
      assertion failures as well as numeric errors in randf.
   2. In the current version, the random seed is not properly
      initialized from the demo file.  It says it does, but it doesn't.
      (Instead a random number is generated from time(0)).

    GOREY DETAILS:  For things to work, the call to init_rand() must be
    delayed until after the seed is read from the demo file. 
    Unfortunately just delaying the init_rand() call till after
    play_demo  caused the "uninitialized random generator" throw bug. To
    get around that problem, the code that reads the random seed was
    moved from GameSession :: play_demo method to a new method: 
    GameSession :: get_demo_random_seed, which could occur earlier. 
    Unfortunately in the process of doing that, play_demo stopped
    skipping over the initial seed, got confused, and prevented
    play_demo from operating at all. All of this should be fixed now.

Other lesser changes that are also in this patch are:

   1. RandomGenerator::rand(int) now has no fence-post biases. 
      Previously,  there were large biases in cases like when it's
      argument was, say, =rand_max*0.75.
   2. A debug mode was added to RandomGenerator, but set to OFF.
   3. A periodic call of rand had been added every 20 frames to aid in
      debug.  It was changed to call rand every cycle because the "every
      20 times" functionality broke due to other's changes.  The call is
      currently commented out, but I un-comment it for debug.
   4. The random_seed stored in the demo file is saved as decimal with
      leading spaces. This removes a trivial debugging cut-and-paste gotcha.
   5. in random_generator, an assertion check (of sizeof(int) >= 4) was
      moved into initialization code to reduce processing time.
   6. An old commented-out erroneous call to srand(0) was removed totally.

Date: 2006-Jun-27 20:18

Comment:
This problem is related to AMD64 processors and caused because the command line arguments supplied to jam's linker are too long for jam to handle.  A temporary workaround to this problem is to run the configure script with "--build=x86_64-pc-linux-gnu":
./configure --build=x86_64-pc-linux-gnu

This will shorten the list of arguments down by a few hundred characters, bringing you back under the 10240 limit.  Note that you should run "jam clean" before reconfiguring and rebuilding.

>
>  It is based on a recent version (3821), and effects the following changes.
>
>
> The variable RAND_MAX is machine-dependent, and shouldn't be used inside the
> machine-independent RandomGenerator, as it currently is.  This patch fixes
> that problem, which expressed itself in Wintel platforms (where
> RAND_MAX=0x7fff and INT_MAX=0x7fffffff) as assertion failures as well as
> numeric errors in randf.
> In the current version, the random seed is not properly initialized from the
> demo file.  It says it does, but it doesn't. (Instead a random number is
> generated from time(0)).
>
> GOREY DETAILS:  For things to work, the call to init_rand() must be delayed
> until after the seed is read from the demo file.  Unfortunately just
> delaying the init_rand() call till after play_demo  caused the
> "uninitialized random generator" throw bug. To get around that problem, the
> code that reads the random seed was moved from GameSession :: play_demo
> method to a new method:  GameSession :: get_demo_random_seed, which could
> occur earlier.  Unfortunately in the process of doing that, play_demo
> stopped skipping over the initial seed, got confused, and prevented
> play_demo from operating at all. All of this should be fixed now.
>
>  Other lesser changes that are also in this patch are:
>
>
> RandomGenerator::rand(int) now has no fence-post biases.  Previously,  there
> were large biases in cases like when it's argument was, say, =rand_max*0.75.
>
> A debug mode was added to RandomGenerator, but set to OFF.
> A periodic call of rand had been added every 20 frames to aid in debug.  It
> was changed to call rand every cycle because the "every 20 times"
> functionality broke due to other's changes.  The call is currently commented
> out, but I un-comment it for debug.
> The random_seed stored in the demo file is saved as decimal with leading
> spaces. This removes a trivial debugging cut-and-paste gotcha.
> in random_generator, an assertion check (of sizeof(int) >= 4) was moved into
> initialization code to reduce processing time.
>
> An old commented-out erroneous call to srand(0) was removed totally.

Hello SuperTux Crew.

We have gone back and forth over the last year with respect to making the
Tux sprite a single image again as opposed to the current four section
body parts. If anyone here on the ML was around when the split was made, -
feel free to shed light on why this split was a good idea or why this was
done.

In discussions on IRC I was commenting on how I would like to basically
zero out the head and feet and move towards a two sprite solution with a 
body and an arm sprites. From the graphic side this seems like the easiest
way to get some relief from the complexities of the current sectional Tux,
while still retaining a normal look as Tux's carried object gets layered
between the arms and the body.

IIRC, the other day Grumble had commented in favor of just a single Tux
sprite.  With additional single graphics for every situation.
i.e. - BigTux, BigTux with Block, FirehatTux, FirehatTux with Block, etc..

I started the process of making xcf templates for each 'singleTux'
situation but now I have questions about wether this is a good idea. The
problem I see is how natural things look and work, especially with regard
to holding an object and jumping through a 1 tile hole.

On IRC today I discussed this with Sommer and his comments were:
"I don't know if one complete animation set per type of thing carried is a
good idea. I'd prefer a generic 'Tux carrying something' animation, which
the game can then overlay with e.g. MrIceblock. The game can then let Tux
carry MrIceblock closer to his body when he drops through a 1-tile-hole."

Please speak up if you have comments on how/why the Tux sprite should work.
I am leaning back towards my original game plan of body plus arm sprites.

- greetings :p

Message-ID: <ff70c5070607230645u2afe00c4oc9676a27925d3fc1@mail.gmail.com>

This patch is essentially what I did to the objects in order to try to
improve scripting.  The scripting isn't mergeable because it has
crashing issues, so I tried to make a mergeable patch that only did my
changes to the object system with somewhat minimal changes to the
scripting system.

1.  Expanded portable interface to include a boolean "portable", as
well as support methods and constructors.
2.  Expanded GameObject to include a std::string "name", and added
support methods and constructors, notably one to allow the field to be
acquired from a level file.
3.  Expanded MovingObject to include a boolean "solid", and added
support for what would become a scripting interface.
4.  Removed the "flags" field from GameObject, since it is pointless
with changes 1 and 3.  Fixed where things broke due to its removal.
5.  Made AmbientSound inherit from MovingObject instead of GameObject,
since most things that can actually exist physically in a level should
be MovingObjects, and I know how that class works.
6.  Loosened the singletoness of Camera, DisplayEffect, and Player.
Level doesn't have a host object, so I'm not sure how to handle that.

SuperTux Milestone 1 did not really have something like "lives". Just 
like now, you could always continue your journey from the start of the 
last level you played. This has made it the perfect causal-gamer's toy 
to play with during short breaks and contributed to its huge success (or 
so I believe).

This concept, however, makes keeping something like "score" pretty much 
impossible. With unlimited tries and retries for any given level, you 
can reach an arbitrary number of points, so we just dropped it.

I never look at the score in any game I play, except for where the
entire point of the game is the score (eg, pinball). For Supertux the
goal is to finish the game, not to get a high score, imo. However, we
do have time trials and coin trials that should add to the
replayability of the game.

One of the biggest problems with scores, I think, is how to calculate
it. It's tough to come up with a formula that takes into account
badguys killed, coins collected and level time. If the formula ever
needs to be changed, it basically voids all scores from previous
versions of the game. I think this was one of the main reasons we
decided to do away with a score.
